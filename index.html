SUBTEMA 1.1

Árboles de Expresiones

1
¿Qué es? Un árbol de expresiones representa una expresión aritmética/lógica como estructura jerárquica donde los nodos internos son operadores (+, -, *, /) y las hojas son operandos.
2
Ejemplo base: La expresión (3 + 5) * 2 se convierte en un árbol donde la raíz es *, su hijo izquierdo es el subárbol + con hojas 3 y 5, y el derecho es 2.
3
Evaluación postorden: Se recorre izquierda → derecha → raíz. Primero se evalúan los subárboles y luego se aplica el operador del nodo padre.
4
Utilidad: Los compiladores usan estos árboles para generar código intermedio, optimizar expresiones y aplicar reglas algebraicas de simplificación.
Expresión: (3 + 5) * 2
* / \ + 2 / \ 3 5
Expresión: a * b + c / d
+ / \ * / / \ / \ a b c d
Expresión: (x + 1) * (y - 2)
* / \ + - / \ / \ x 1 y 2
SUBTEMA 1.2

Árboles Semánticos de Analizadores Sintácticos

1
AST vs. árbol de análisis: El parse tree incluye todos los símbolos gramaticales. El AST lo simplifica eliminando tokens auxiliares como paréntesis y separadores.
2
Nodos: Cada nodo representa una construcción: declaración, asignación, llamada a función, condicional, etc. Los atributos semánticos (tipo, valor, ámbito) se almacenan en los nodos.
3
Construcción: En parsers descendentes los nodos se crean durante la expansión de reglas; en ascendentes, al reducirlas.
4
Uso posterior: El AST es la entrada para verificación de tipos, resolución de nombres y generación de código intermedio.
Sentencia: int x = 3 + y;
Asignación / \ Declaración BinOp(+) (int, "x") / \ Lit(3) Var(y)
Sentencia: if (a > b) x = 1;
If / \ BinOp(>) Asign / \ / \ Var(a) Var(b) Var(x) Lit(1)
Llamada: resultado = suma(a, b);
Asignación / \ Var(resultado) LlamadaFn / | \ "suma" Var(a) Var(b)
SUBTEMA 1.3

Comprobaciones de Tipos de Expresiones

1
Estático vs. dinámico: En tipado estático (C, Java) los tipos se verifican en compilación. En dinámico (Python, JS) en ejecución. Ambos usan reglas de compatibilidad.
2
Reglas de tipo: Para un BinOp(+), si ambos operandos son int el resultado es int; si uno es float, el resultado es float (coerción implícita).
3
Error de tipo: Si se opera con tipos incompatibles (ej. int + string en C), el analizador semántico emite un error antes de generar código.
4
Tabla de tipos: Se usa junto a la tabla de símbolos para registrar el tipo declarado de cada variable y compararlo contra su uso en expresiones.
Inferencia: int + float → float
BinOp(+) → float / \ Var(a):int Var(b):float ↑ coerción implícita
Error: int + string
BinOp(+) → ❌ ERROR SEMÁNTICO / \ Var(x):int Var(s):string ↑ tipos incompatibles
Asignación con conversión
Asign(z:float) | BinOp(*) → int / \ Lit(3):int Lit(4):int ↑ int→float al asignar
SUBTEMA 1.4

Pila Semántica en un Analizador Léxico

1
¿Qué es? Es una estructura LIFO que mantiene los atributos semánticos (tipo, valor, posición) asociados a los tokens mientras el analizador los procesa.
2
PUSH: Cada vez que el scanner reconoce un token, empuja su atributo semántico. Al reconocer 42, empuja {tipo:INT, valor:42}.
3
POP en reducción: Cuando el parser reduce una producción, extrae los atributos y calcula el atributo resultante para el no-terminal izquierdo.
4
En yacc/bison: Se gestiona automáticamente vía $$ (valor del no-terminal) y $1, $2, ... (valores de la producción).
Proceso: análisis de "3 + 5"
Token leído Acción Pila semántica ───────────────────────────────────────────── "3" PUSH [ {INT,3} ] "+" PUSH [ {INT,3} | {OP,+} ] "5" PUSH [ {INT,3} | {OP,+} | {INT,5} ] (reducir) POP×3, PUSH [ {INT,8} ] $$ = $1 + $3 = 3 + 5 = 8
Estado de pila: int x = 10
Token Pila (tope→base) ────────────────────────────── int [ {TIPO,"int"} ] x [ {ID,"x"} | {TIPO,"int"} ] = [ {OP,"="} | {ID,"x"} | {TIPO,"int"} ] 10 [ {INT,10} | {OP,"="} | {ID,"x"} | {TIPO,"int"} ] ; Reducir: declarar x:int = 10 → POP×4
SUBTEMA 1.5

Esquema de Traducción

1
¿Qué es? Un esquema de traducción (SDT) es una gramática libre de contexto con acciones semánticas incrustadas entre los símbolos de sus producciones. Cada acción se ejecuta cuando el parser la alcanza durante el análisis.
2
Acciones incrustadas: Se escriben entre llaves { } en una producción. Ejemplo: E → E₁ + T { E.val = E₁.val + T.val }. La acción calcula el atributo del no-terminal resultante al momento de la reducción.
3
Atributos sintetizados vs. heredados: Los sintetizados fluyen de hijos a padres (bottom-up). Los heredados fluyen de padres a hijos (top-down). Los SDT pueden manejar ambos tipos de atributos.
4
Generación de código: Las acciones emiten instrucciones de código de tres direcciones. Al reducir E → E₁ + T se puede generar la cuádrupla (+, E₁.place, T.place, t1).
Producción con acción semántica
E → E₁ + T { E.val = E₁.val + T.val } ↑ acción ejecutada al reducir esta producción Traza con entrada "3 + 5": ────────────────────────────────────── T.val = 3 E₁.val = 3 T.val = 5 Acción → E.val = 3 + 5 = 8 ✓
SDT completo para expresión aritmética
E → E + T { E.val = E₁.val + T.val; emit('+', E₁.place, T.place, E.place) } E → E - T { E.val = E₁.val - T.val; emit('-', E₁.place, T.place, E.place) } E → T { E.val = T.val } T → T * F { T.val = T₁.val * F.val; emit('*', T₁.place, F.place, T.place) } T → F { T.val = F.val } F → ( E ) { F.val = E.val } F → num { F.val = num.lexval }
Código de 3 direcciones generado para: a + b * c
Cuádruplas emitidas (op, arg1, arg2, resultado): ──────────────────────────────────────────────── ( * , b , c , t1 ) → t1 = b * c ( + , a , t1 , t2 ) → t2 = a + t1 ──────────────────────────────────────────────── Resultado final almacenado en: t2
SUBTEMA 1.6

Tabla de Símbolo y Tabla de Direcciones

1
Tabla de Símbolos: Estructura de datos (generalmente una tabla hash) que almacena información sobre cada identificador: nombre, tipo, ámbito (scope), línea de declaración y categoría (variable, función, constante, parámetro).
2
Tabla de Direcciones: Complementa la tabla de símbolos asignando una dirección de memoria relativa (offset) a cada variable, indicando al generador de código dónde almacenar y recuperar cada valor en el frame de pila.
3
Gestión de ámbitos: Se usan pilas de tablas anidadas. Al entrar a una función se crea una nueva tabla local; al salir se destruye, permitiendo reutilizar nombres en distintos bloques sin colisión.
4
Consultas en compilación: Cada referencia a un identificador consulta la tabla desde el ámbito más interno hacia el global. Si no existe → error "variable no declarada". Si existe → se obtiene tipo y dirección para generar código.
Tabla de Símbolos — programa ejemplo
Nombre │ Tipo │ Ámbito │ Línea │ Categoría ──────────┼─────────┼──────────┼───────┼─────────── x │ int │ global │ 1 │ variable y │ float │ global │ 2 │ variable PI │ float │ global │ 3 │ constante suma │ int │ global │ 5 │ función a │ int │ suma │ 5 │ parámetro b │ int │ suma │ 5 │ parámetro resultado │ int │ suma │ 6 │ variable
Tabla de Direcciones — offset en el frame
Nombre │ Tipo │ Tamaño │ Offset ──────────┼─────────┼─────────┼──────── x │ int │ 4 bytes │ 0 y │ float │ 4 bytes │ 4 resultado │ int │ 4 bytes │ 8 ───────────────────────────────────────────── Dirección base del frame: 1000 Dir. real de y = 1000 + 4 = 1004 Dir. real de resultado = 1000 + 8 = 1008
Pila de ámbitos durante ejecución de suma()
┌──────────────────────────────────┐ │ Tabla local: a, b, resultado │ ← ámbito activo ├──────────────────────────────────┤ │ Tabla global: x, y, PI, suma │ ← ámbito base └──────────────────────────────────┘ Al salir de suma() → tabla local destruida
SUBTEMA 1.7

Manejo de Errores Semánticos

1
¿Qué son? Errores de significado que superan el análisis sintáctico pero violan las reglas del lenguaje: variable no declarada, tipos incompatibles, número incorrecto de argumentos, retorno con tipo inválido, etc.
2
Detección: El analizador semántico recorre el AST consultando la tabla de símbolos. Si encuentra una referencia a un identificador inexistente o una operación inválida entre tipos, registra el error con línea, columna y descripción.
3
Recuperación: El compilador no debe detenerse en el primer error. Estrategia: asignar tipo genérico error al nodo problemático para que no propague falsos positivos, insertar un valor por defecto y continuar el análisis del resto del programa.
4
Reporte efectivo: Un buen compilador muestra línea, columna, tipo de error y sugerencia. Ejemplo: línea 5: variable 'z' no declarada — ¿quisiste decir 'x'?. Esto acelera la corrección del programador.
Clasificación de errores semánticos comunes
Tipo de error │ Ejemplo en código C ────────────────────────────┼────────────────────────────────── Variable no declarada │ x = 5; // 'x' no existe Tipo incompatible │ int a = "hola"; // string → int ✗ Argumentos incorrectos │ suma(1, 2, 3); // suma espera 2 args Redeclaración en mismo ámbito│ int x; int x; // 'x' ya existe Retorno con tipo inválido │ int f(){ return "ok"; } Uso antes de declaración │ y = x + 1; int x; // x no disponible aún
Propagación del tipo error — evita falsos positivos
BinOp(+) / \ Var(z):❌ Lit(3):int ↑ 'z' no declarada → se asigna tipo: error Nodo padre también hereda tipo: error → El compilador NO genera "operación int+error inválida" porque ya reportó el error raíz en z. → Esto evita cadenas de errores derivados.
Mensajes de error bien formateados
[Error semántico] línea 3, col 5: → Variable 'z' usada sin declarar. [Error semántico] línea 7, col 12: → Tipo incompatible: no se puede asignar 'string' a 'int'. → Sugerencia: usa atoi() para convertir el valor. [Advertencia] línea 10, col 8: → Variable 'resultado' declarada pero nunca utilizada.
SUBTEMA 2.1

Notaciones

1
Notación Prefija: El operador aparece antes de sus operandos. Ejemplo: + 3 5 equivale a 3 + 5. No requiere paréntesis ni reglas de precedencia para su evaluación.
2
Notación Infija: El operador aparece entre sus operandos. Es la notación matemática habitual: 3 + 5. Requiere paréntesis o reglas de precedencia para evitar ambigüedades.
3
Notación Postfija (RPN): El operador aparece después de sus operandos. Ejemplo: 3 5 +. Los compiladores la prefieren porque se evalúa fácilmente con una pila, sin necesidad de paréntesis.
4
Conversión: Para convertir infija a postfija se usa el algoritmo Shunting-yard de Dijkstra, que emplea una pila de operadores y respeta la precedencia y asociatividad.
Comparación de las tres notaciones para: (3 + 5) * 2
Infija → (3 + 5) * 2 Prefija → * + 3 5 2 Postfija → 3 5 + 2 *
Evaluación postfija de "3 5 + 2 *" con pila
Token Acción Pila ───────────────────────────────── 3 PUSH [ 3 ] 5 PUSH [ 3, 5 ] + POP 5,3 → 8 [ 8 ] 2 PUSH [ 8, 2 ] * POP 2,8 → 16 [ 16 ] ───────────────────────────────── Resultado: 16
Árbol con las tres notaciones mapeadas
*(prefija: primer nodo) / \ + 2 / \ 3 5 Prefija: * + 3 5 2 (preorden) Postfija: 3 5 + 2 * (postorden) Infija: (3+5)*2 (inorden)
SUBTEMA 2.2

Representaciones de Código Intermedio

1
Notación Polaca (RPN): Representación postfija de expresiones. Usada internamente por intérpretes y compiladores para evaluar expresiones con una pila sin necesidad de paréntesis.
2
Código P: Código intermedio para una máquina de pila abstracta (P-machine). Cada instrucción opera sobre una pila. Usado históricamente por compiladores de Pascal.
3
Triplos: Representación con tres campos: (operador, arg1, arg2). El resultado se referencia por número de instrucción. No requiere nombres temporales explícitos, pero son difíciles de reordenar.
4
Cuádruplos: Representación con cuatro campos: (op, arg1, arg2, resultado). El resultado se almacena en una variable temporal explícita (t1, t2…), facilitando la optimización y reordenamiento.
Expresión: a = b * c + d — en las 4 representaciones
NOTACIÓN POLACA (postfija): b c * d + a = CÓDIGO P: LOAD b LOAD c MUL LOAD d ADD STORE a
Triplos
No. │ Op │ Arg1 │ Arg2 ─────┼─────┼──────┼────── (1) │ * │ b │ c (2) │ + │ (1) │ d ← referencia a triplo 1 (3) │ = │ a │ (2) ← referencia a triplo 2
Cuádruplos
No. │ Op │ Arg1 │ Arg2 │ Result ─────┼─────┼──────┼──────┼──────── (1) │ * │ b │ c │ t1 (2) │ + │ t1 │ d │ t2 (3) │ = │ t2 │ — │ a
SUBTEMA 2.3

Esquema de Generación

1
Variables y constantes: Una variable genera una instrucción LOAD (cargar su dirección). Una constante genera LDC (load constant). En cuádruplos: (=, valor, —, t1).
2
Expresiones: Se recorre el AST en postorden. Cada nodo BinOp genera un cuádruplo (op, t_izq, t_der, t_nuevo). Los temporales se van acumulando hasta obtener el resultado final.
3
Instrucciones de control: Los if-else generan etiquetas y saltos condicionales. Ejemplo: (IF_FALSE, cond, —, L1), (GOTO, —, —, L2), (LABEL, L1).
4
Funciones y estructuras: Las llamadas generan secuencias PARAM + CALL. Las estructuras (structs/arreglos) se acceden con aritmética de offsets: (+, base, offset, dir).
Asignación: x = a + b * c
Cuádruplos generados: ───────────────────────────────────────── (1) ( * , b , c , t1 ) → t1 = b * c (2) ( + , a , t1 , t2 ) → t2 = a + t1 (3) ( = , t2 , — , x ) → x = t2
Control: if (a > b) x = 1; else x = 2;
(1) ( > , a , b , t1 ) (2) ( IF_FALSE, t1 , — , L1 ) (3) ( = , 1 , — , x ) (4) ( GOTO , — , — , L2 ) (5) ( LABEL , L1 , — , — ) (6) ( = , 2 , — , x ) (7) ( LABEL , L2 , — , — )
Llamada a función: resultado = suma(a, b)
(1) ( PARAM , a , — , — ) (2) ( PARAM , b , — , — ) (3) ( CALL , suma , 2 , t1 ) (4) ( = , t1 , — , resultado)
SUBTEMA 3.1

Tipos de Optimización

1
Locales: Se aplican dentro de un bloque básico (secuencia lineal sin saltos). Incluyen eliminación de subexpresiones comunes, propagación de constantes y simplificación algebraica.
2
De ciclos: Se aplican en bucles (for, while). Técnicas: movimiento de código invariante (sacar cálculos del bucle), reducción de fuerza (reemplazar multiplicaciones por sumas) y eliminación de variables de inducción.
3
Globales: Análisis inter-bloques sobre el grafo de flujo completo del programa. Incluyen análisis de alcanzabilidad de definiciones, variables vivas y propagación global de constantes.
4
De mirilla (Peephole): Se examina una ventana pequeña de instrucciones adyacentes y se reemplaza por secuencias más cortas o eficientes. Ejemplo: eliminar STORE x / LOAD x consecutivos.
Optimización local — eliminación de subexpresiones comunes
ANTES: DESPUÉS: ────────────────────────────────────────── t1 = b * c t1 = b * c t2 = b * c ←duplicada t2 = a + t1 t3 = a + t1 t3 = t2 ← reutiliza t2 t4 = a + t1 ←duplicada
Optimización de ciclo — código invariante
ANTES: DESPUÉS: ──────────────────────────────────────── for i = 1 to n: t1 = x * y ← sacado del bucle a[i] = x * y + i for i = 1 to n: a[i] = t1 + i
Optimización de mirilla (peephole)
ANTES: DESPUÉS: ────────────────────────────────── STORE x (instrucción eliminada) LOAD x → (instrucción eliminada) ADD y ADD y ← el valor ya está en registro
SUBTEMA 3.2

Costos

1
Costo de ejecución: Se mide en accesos a memoria, uso de registros y operaciones de pila. El objetivo es maximizar el uso de registros (rápidos) y minimizar accesos a memoria RAM (lentos).
2
Criterios para mejorar el código: Reducir instrucciones totales, minimizar saltos (afectan el pipeline), reutilizar resultados ya calculados en registros, y preferir operaciones de menor ciclo de reloj.
3
Análisis de flujo de datos: Técnica que calcula propiedades del programa en cada punto del grafo de flujo. Permite saber qué variables están vivas (liveness), qué definiciones alcanzan un punto (reaching definitions) y detectar código muerto.
4
Herramientas: Grafos de flujo de control (CFG), árboles de dominadores, análisis de alias y herramientas como Valgrind, gprof o el optimizador de LLVM (opt) con sus distintos niveles -O1 / -O2 / -O3.
Comparativa de costo por tipo de instrucción (ciclos de reloj aproximados)
Operación │ Costo (ciclos) ───────────────────────┼──────────────── Registro → Registro │ 1 Suma/Resta entera │ 1 Multiplicación entera │ 3 – 5 Acceso a caché L1 │ 4 Acceso a RAM │ 100 – 300 Salto condicional │ 1 – 20 (depende de predicción) División entera │ 20 – 40
Análisis de liveness — variables vivas
Instrucción Variables vivas DESPUÉS ──────────────────────────────────────────── (1) t1 = b * c { b, c, a, t1 } (2) t2 = a + t1 { t2 } (3) x = t2 { } ← x no se usa después → código muerto posible Si x nunca se lee tras esta asignación: → Las instrucciones (1)(2)(3) pueden eliminarse.
